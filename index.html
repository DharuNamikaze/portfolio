Every startup eventually builds what I call a mini-AWS inside their own infrastructure. It starts innocently. You need caching — spin up Redis. Search — add Elasticsearch. Your AI feature needs vector search — hello Pinecone. The RAG pipeline needs orchestration — here comes LangChain. Geo search? InfluxDB? Multi-tenant auth?

Eleven services. Eleven SDKs. Eleven failure modes. Eleven bills. Eleven things your new teammate has to learn before they can ship anything. And you reached for each one of them before you'd actually hit the limit of your database.

Postgres has been quietly building answers to every one of these problems for years. But most "use Postgres for everything" posts stop there — they tell you that you can, not until when you can. This post fixes that. Every replacement comes with a concrete threshold so you know exactly when to migrate and what breaks first.

SCOPE This is architecture-level guidance for growth-phase SaaS — roughly the 0 to 500k MAU range. At massive scale, dedicated systems win. The argument here is that you probably aren't there yet, and the complexity cost of fragmented infrastructure is very real.
Kill #1

Replacing Redis

Redis earns its keep for three things: key-value caching, job queues, and pub/sub. Postgres has solid — though not identical — answers for all three.

1. Key-value caching with unlogged tables

Unlogged tables skip the write-ahead log entirely. They're crash-unsafe on restart, but dramatically faster for ephemeral data. Pair with a ttl column and a pg_cron cleanup job and you have Redis-style caching without leaving your DB. 

CREATE UNLOGGED TABLE cache (
  key   TEXT        PRIMARY KEY,
  value JSONB       NOT NULL,
  ttl   TIMESTAMPTZ NOT NULL DEFAULT now() + '1 hour'
);

-- Upsert a cache entry
INSERT INTO cache (key, value)
VALUES ('user:42:profile', '{"name":"Dharu"}'::jsonb)
ON CONFLICT (key) DO UPDATE SET
  value = EXCLUDED.value,
  ttl   = now() + '1 hour';

-- Fetch (null if expired)
SELECT value FROM cache
WHERE key = 'user:42:profile' AND ttl > now();

-- Evict stale entries every 5 min via pg_cron
DELETE FROM cache WHERE ttl < now();
Latency ceiling Unlogged tables can handle the volume but not in-memory latency. Postgres RTT is 1–5ms. Redis is sub-millisecond. For high-frequency reads (social feeds, hot session paths), you'll feel this difference. The signal to watch is p99 cache read latency crossing 5ms — not request throughput. Rule of thumb: keep Redis if you need <1ms reads at >15,000 ops/sec sustained.

2. Job queues with SKIP LOCKED

SELECT FOR UPDATE SKIP LOCKED lets multiple workers pull jobs concurrently without contention. This is how Que, GoodJob, and Oban work — it's a production-proven pattern, not a hack. 

CREATE TABLE job_queue (
  id         BIGSERIAL PRIMARY KEY,
  queue_name TEXT        NOT NULL DEFAULT 'default',
  payload    JSONB       NOT NULL,
  status     TEXT        NOT NULL DEFAULT 'pending',
  run_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Worker: claim exactly one job atomically
WITH next_job AS (
  SELECT id FROM job_queue
  WHERE status = 'pending' AND run_at <= now()
  ORDER BY run_at
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
UPDATE job_queue SET status = 'processing'
WHERE id = (SELECT id FROM next_job)
RETURNING *;
Threshold Works well up to ~500 jobs/sec. Beyond ~2,000 jobs/sec, row lock churn becomes measurable — watch pg_stat_activity for worker contention. For email, webhooks, async tasks at SaaS scale: this is more than enough.

3. Pub/sub with LISTEN / NOTIFY

NOTIFY sends a lightweight signal over an existing connection. LISTEN subscribes to it. Perfect for cache invalidation — terrible for anything requiring delivery guarantees. 

-- Send a signal when a product changes
CREATE OR REPLACE FUNCTION notify_product_change()
RETURNS TRIGGER AS $$
BEGIN
  PERFORM pg_notify('product_updates', row_to_json(NEW)::text);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER product_change_trigger
AFTER INSERT OR UPDATE ON products
FOR EACH ROW EXECUTE FUNCTION notify_product_change();

-- Subscribe in your app
-- await client.query('LISTEN product_updates');
-- client.on('notification', (msg) => cache.invalidate(...));
Hard limit NOTIFY has no persistence. If your consumer is down, the message is gone. No replay, no fan-out to multiple consumers, no delivery guarantee. Use it only for stateless signals (cache bust, UI refresh). For anything requiring guaranteed delivery: keep a queue table or reach for a real message broker.
Kill #2

Replacing Elasticsearch

Elasticsearch is a beast to operate. Postgres's full-text search won't beat it at petabyte log ingestion, but for product search, blog search, and content discovery it's more than sufficient — and it stays transactionally consistent with your main data.

tsvector + tsquery + GIN index

-- Generated search column — no manual sync needed
ALTER TABLE products
ADD COLUMN search_vector TSVECTOR
GENERATED ALWAYS AS (
  setweight(to_tsvector('english', coalesce(name, '')), 'A') ||
  setweight(to_tsvector('english', coalesce(description, '')), 'B') ||
  setweight(to_tsvector('english', coalesce(brand, '')), 'C')
) STORED;

CREATE INDEX idx_products_fts
  ON products USING GIN (search_vector);

-- Search with relevance ranking
SELECT id, name, price,
  ts_rank(search_vector, query) AS rank
FROM products,
  websearch_to_tsquery('english', 'wireless noise cancelling headphones') query
WHERE search_vector @@ query
ORDER BY rank DESC LIMIT 20;

websearch_to_tsquery() understands AND, OR, and quoted phrases — users type naturally and it works. Add pg_trgm for fuzzy matching on typos.

Real ceiling — relevance, not document count The honest limit is relevance tuning, not the number of documents. BM25 scoring, custom analyzers, synonyms, and learning-to-rank features — Elasticsearch wins there. Document count is secondary: Postgres FTS is fast up to ~10–20M docs; at 50M+ with complex filter combinations, p99 starts drifting. But if you hit relevance tuning as a product requirement before you hit 10M documents, consider moving earlier.
Kill #3

Replacing Pinecone

pgvector brings vector similarity search directly into Postgres. The killer advantage: your embeddings live in the same database as your content — no sync job, no divergence, atomic joins between vector results and structured filters. 

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE document_chunks (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  content   TEXT         NOT NULL,
  metadata  JSONB,
  embedding VECTOR(1536),  -- OpenAI text-embedding-3-small
  created_at TIMESTAMPTZ DEFAULT now()
);

-- HNSW index (faster queries, ideal for most use cases)
CREATE INDEX idx_chunks_hnsw
  ON document_chunks USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- Hybrid: vector similarity + metadata filter in one query
SELECT id, content, embedding <=> $1 AS dist
FROM   document_chunks
WHERE  metadata ->> 'source' = 'product_docs'
ORDER BY dist LIMIT 5;
RAM is your ceiling, not row count The HNSW index lives in RAM. At 1536 dimensions, 1M vectors ≈ 6GB of index memory. At 5M vectors you're at ~30GB. The moment your HNSW index starts getting evicted from RAM, queries slow dramatically. Watch memory pressure, not vector count. At ~20M+ vectors, dedicated ANN systems (Qdrant, Weaviate) win because their infrastructure is tuned for this.
<-> Euclidean

L2 distance. Best for image embeddings and geometric similarity.

<=> Cosine

Best for text embeddings. Insensitive to vector magnitude.

<#> Inner product

Fast with pre-normalized embeddings. Used in some recommendation systems.

Atomic joins

Vector results + SQL filters in a single query. Impossible with a separate vector DB.

Kill #4

Replacing LangChain

LangChain is a framework for chaining LLM calls: retrieve context, build a prompt, call the model, parse the output. It's powerful but notoriously heavy. At its core, a RAG pipeline is a few HTTP requests and some string manipulation. With pgvector and the http extension, you can do this entirely in Postgres. 

CREATE EXTENSION IF NOT EXISTS http;

CREATE OR REPLACE FUNCTION rag_query(
  user_question   TEXT,
  query_embedding VECTOR(1536),
  top_k           INT DEFAULT 5
) RETURNS TEXT AS $$
DECLARE
  context_chunks TEXT;
  prompt         TEXT;
  api_response   http_response;
  answer         TEXT;
BEGIN
  -- Step 1: retrieve semantically relevant chunks
  SELECT string_agg(content, E'\n---\n')
  INTO context_chunks
  FROM (
    SELECT content FROM document_chunks
    ORDER BY embedding <=> query_embedding LIMIT top_k
  ) c;

  -- Step 2: build the prompt
  prompt := format(
    'Answer using only this context:\n\n%s\n\nQuestion: %s',
    context_chunks, user_question
  );

  -- Step 3: call the LLM
  SELECT * INTO api_response
  FROM http_post(
    'https://api.openai.com/v1/chat/completions',
    json_build_object('model', 'gpt-4o-mini', 'messages',
      json_build_array(json_build_object('role', 'user', 'content', prompt))
    )::text, 'application/json'
  );

  -- Step 4: parse and return
  SELECT api_response.content::jsonb
    -> 'choices' -> 0 -> 'message' ->> 'content'
  INTO answer;
  RETURN answer;
END;
$$ LANGUAGE plpgsql;
When LangChain still wins The PL/pgSQL approach is elegant for simple retrieval → generate flows. Multi-agent loops, streaming responses, complex tool-use chains, and retry logic all benefit from a proper framework. Use Postgres RAG for the 80% case; reach for LangChain or LlamaIndex when your product requires complex orchestration logic.
Kill #5

Replacing InfluxDB — Time-series with TimescaleDB

TimescaleDB is a Postgres extension that adds automatic time-based partitioning, time-series aggregation functions, and continuous aggregates. It looks like a normal Postgres table to your app. 

CREATE EXTENSION IF NOT EXISTS timescaledb;

CREATE TABLE metrics (
  time    TIMESTAMPTZ      NOT NULL,
  service TEXT             NOT NULL,
  metric  TEXT             NOT NULL,
  value   DOUBLE PRECISION,
  tags    JSONB
);
SELECT create_hypertable('metrics', 'time',
  chunk_time_interval => INTERVAL '1 week');

-- P99 response time bucketed by 5 minutes
SELECT
  time_bucket('5 minutes', time) AS bucket,
  service,
  percentile_cont(0.99) WITHIN GROUP (ORDER BY value) AS p99
FROM  metrics
WHERE metric = 'response_time_ms'
  AND time > now() - INTERVAL '1 hour'
GROUP BY bucket, service
ORDER BY bucket DESC;
Threshold: ~100k rows/sec write pressure TimescaleDB is great for product analytics, APM, and IoT at reasonable scale. The gap versus dedicated TSDBs is columnar compression — ClickHouse can store the same data in 10× less space and query it 10× faster at the high end. Watch WAL write pressure on your primary. If metrics ingestion is causing WAL lag that affects your main transactional workload, time to split.
Kill #6

Replacing Google Maps API — PostGIS

Every call to the Google Maps Distance Matrix API or Nearby Places API is a paid request. PostGIS gives you the same capabilities at zero marginal cost: radius search, polygon containment, distance calculation — all in SQL. 

CREATE EXTENSION IF NOT EXISTS postgis;

CREATE TABLE stores (
  id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name     TEXT                   NOT NULL,
  location GEOGRAPHY(POINT, 4326)  -- WGS84 lat/lng
);

CREATE INDEX idx_stores_location
  ON stores USING GIST (location);

-- Stores within 10km — with distance in the result
SELECT name,
  ST_Distance(location, ST_MakePoint(80.2707, 13.0827)::geography) / 1000
  AS distance_km
FROM   stores
WHERE  ST_DWithin(location, ST_MakePoint(80.2707, 13.0827)::geography, 10000)
ORDER BY distance_km;

-- Delivery zone — is this customer inside the polygon?
SELECT dz.name
FROM   delivery_zones dz
WHERE  ST_Contains(dz.zone::geometry, ST_MakePoint(80.2707, 13.0827));
Strongest replacement on this list PostGIS is the most "unlimited" swap here. You'd need Google Maps specifically for forward geocoding (address string → coordinates) — PostGIS doesn't do that. For everything else, it's production-ready at arbitrary scale with a spatial GIST index. Use Nominatim (self-hosted OpenStreetMap) if you need geocoding.
Kill #7

Replacing Casbin / OPA — Row Level Security

Most multi-tenant apps implement authorization in application code. This is fragile — one missed check equals a data leak. Row Level Security enforces access policies at the database layer. It's physically impossible to return rows a user shouldn't see. 

ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

-- Customers see only their own orders
CREATE POLICY orders_isolation ON orders
FOR ALL USING (
  customer_id = current_setting('app.current_user_id')::uuid
);

-- Multi-tenant: each org sees only their products
CREATE POLICY products_tenant_isolation ON products
USING (org_id = current_setting('app.current_org_id')::uuid);

-- Set per-request in your app (Node.js)
-- await client.query(`
--   SET LOCAL app.current_user_id = '${userId}';
--   SET LOCAL app.current_org_id  = '${orgId}';`
-- );
Threshold: policy complexity, not scale RLS breaks when your authorization logic needs to call an external service — "is this user's subscription active in Stripe?", "does this user have a feature flag set in LaunchDarkly?". RLS is database-local; it can't make HTTP calls. Also: RLS adds ~5–10% query overhead. For complex policies with many joins, benchmark before assuming it's free.
Kill #8

Replacing MongoDB — JSONB

The classic reason for MongoDB is "flexible schema." JSONB gives you exactly that, while keeping your data in the same ACID-compliant, transactional database as the rest of your models. 

CREATE TABLE products (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name       TEXT       NOT NULL,
  price_inr  NUMERIC(10,2) NOT NULL,
  attributes JSONB  -- flexible per-category schema
);

-- GIN index: all JSONB operators run fast
CREATE INDEX idx_products_attrs ON products USING GIN (attributes);

-- Containment query
SELECT * FROM products
WHERE attributes @> '{"color":"indigo","fabric":"silk"}';

-- Surgical update — no rewrite of full document
UPDATE products
SET attributes = jsonb_set(attributes, '{color}', '"navy"')
WHERE id = $1;
Second strongest replacement GIN + JSONB at 100M+ rows is still fast. You only lose to MongoDB when you need its native aggregation pipeline syntax, multi-master sharding, or change streams. The probability you need those before you need the ACID guarantees Postgres provides is very low for most SaaS products.
Kill #9

Replacing dbt — Materialized Views

A large chunk of everyday dbt usage is "run this aggregation on a schedule and cache the result." Postgres materialized views + pg_cron handle that natively, with zero external tooling. 

-- Pre-compute revenue by category, refreshed nightly
CREATE MATERIALIZED VIEW mv_revenue_by_category AS
SELECT
  p.category,
  date_trunc('day', o.created_at) AS day,
  count(*)                           AS order_count,
  sum(oi.quantity * oi.unit_price_inr) AS revenue_inr
FROM   orders o
JOIN   order_items oi ON oi.order_id = o.id
JOIN   products p    ON p.id = oi.product_id
WHERE  o.status = 'completed'
GROUP BY p.category, day;

-- Non-blocking refresh (requires unique index)
CREATE UNIQUE INDEX idx_mv_rev_cat_unique
  ON mv_revenue_by_category (category, day);

SELECT cron.schedule('refresh-revenue-mv', '0 1 * * *',
  $$REFRESH MATERIALIZED VIEW CONCURRENTLY mv_revenue_by_category$$
);
Threshold: DAG complexity Works until your view dependency graph gets tangled or refresh takes >2 minutes. REFRESH CONCURRENTLY requires a unique index and still takes a full table scan. dbt wins when you have complex multi-step transformations with lineage tracking, incremental models, or data tests. For dashboard caching and simple rollups: materialized views are completely sufficient.
Kill #10

Replacing Kafka — Logical Replication CDC

Change Data Capture streams every row-level change from your database to downstream consumers. The standard stack is Debezium + Kafka. Postgres has CDC built in via logical replication. 

-- postgresql.conf: wal_level = logical

-- Create a replication slot
SELECT pg_create_logical_replication_slot(
  'my_app_slot', 'pgoutput'
);

-- Publish specific tables
CREATE PUBLICATION mithilla_pub
  FOR TABLE products, orders, inventory;

-- wal2json gives clean JSON change events:
-- {"action":"I","table":"orders","columns":[{"name":"id",...}]}
Danger: WAL accumulation — this one can crash your primary This is the most operationally risky swap on the list. Replication slots hold WAL files until the consumer reads them. If your consumer lags or dies, WAL accumulates unboundedly and can fill disk and crash your primary instance. Always set max_slot_wal_keep_size in postgresql.conf to cap retention — which means accepting that a lagging consumer gets its slot dropped. Use this for 1–2 internal consumers (sync to search index, invalidate cache). For fan-out to many consumers, guaranteed replay, or complex ordering requirements: use Kafka or a managed tool like Sequin.
Kill #11

Replacing Fivetran — Foreign Data Wrappers

FDWs let you query external data sources — other Postgres databases, MySQL, S3 CSVs, REST APIs — as if they were local tables. For simple sync jobs that don't need Fivetran's 300+ connectors, FDWs are a clean solution. 

CREATE EXTENSION IF NOT EXISTS postgres_fdw;

CREATE SERVER legacy_db FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host 'legacy.db.internal', dbname 'warehouse');

IMPORT FOREIGN SCHEMA public
  FROM SERVER legacy_db INTO legacy_schema;

-- JOIN across two databases as if local
SELECT o.id, o.total_inr, l.customer_tier
FROM   orders             o
JOIN   legacy_schema.customers l ON l.email = o.customer_email
WHERE  o.created_at > now() - INTERVAL '7 days';

-- file_fdw: query a CSV as a table
CREATE FOREIGN TABLE raw_pincode_data (
  pincode TEXT, district TEXT, state TEXT
) SERVER flat_files
OPTIONS (filename '/data/india_pincodes.csv', format 'csv', header 'true');
Threshold: WHERE clause pushdown FDW is deceptively simple until the query planner fails to push your WHERE clause to the remote server. When pushdown fails, Postgres fetches the entire remote table locally and filters in memory. On a 50M row table, that's a full network transfer. Always check EXPLAIN output for "Foreign Scan" vs "Remote SQL" — if your filter conditions aren't in the Remote SQL, you have a full scan problem.

Honest limits

When not to use this approach

This post would be dishonest without a clear-eyed look at when to reach for the dedicated tool before you hit any of the thresholds above. Sometimes the requirement itself — not the scale — is the deciding factor.

Migrate immediately if you need any of these
Sub-ms cache latency
If your SLA demands cache reads under 1ms — social feed hot path, session lookup at high frequency — keep Redis. Postgres RTT simply can't match in-memory.
High-throughput time-series
>1M writes/sec? TimescaleDB is impressive, but ClickHouse, InfluxDB, and Prometheus exist for exactly this — columnar compression and write-optimized storage are core to their architecture, not an extension.
Billion-vector ANN
pgvector at 5–20M vectors with HNSW is production-solid. Beyond that, dedicated ANN systems (Qdrant, Weaviate, Milvus) have tuned disk-resident indexes that Postgres can't match.
Complex event streaming
Kafka isn't just CDC — it's fan-out to 100 consumers, guaranteed ordering, offset-based replay, and consumer groups. Logical replication handles one-to-one CDC. It doesn't replace the Kafka topology.
Cross-service authorization
If your authz policy needs to call Stripe, check feature flags in LaunchDarkly, or query another microservice — RLS can't do that. OPA or Casbin are the right tools for cross-service policy graphs.
Complex LLM orchestration
The PL/pgSQL RAG function is elegant for simple retrieve → generate flows. Multi-agent loops, streaming responses, tool-use chains with retries, and structured output parsing benefit from a real framework.

The actual decision framework for your SaaS

The real question isn't "when does Postgres break?" — it's "when does the ops complexity of adding a service actually pay off versus the cognitive overhead it adds?" Here's the heuristic that holds up in practice:

Anti-pattern: premature fragmentation
  • Adding Redis at 1k users "because everyone does"
  • Running Kafka before you have more than one consumer
  • Pinecone before you have 100k vectors
  • Elasticsearch before your search has a relevance problem
  • Eleven services, eleven failure modes
The pragmatic path
  • Under 50k MAU: Postgres-everything. Fewer failure modes, one backup strategy, atomic transactions across all data.
  • 50k–500k MAU: monitor the specific metrics in the table above. Migrate the piece that's actually hurting.
  • 500k+ MAU: you'll have dashboards showing the bottleneck. By then you can afford the dedicated tool and the engineer who knows it.

Is Postgres the right tool for every problem at every scale? No. But it's the right starting point for almost all of them. Reach for the dedicated tool when you've hit a real, measured ceiling — not because "that's what everyone uses."
